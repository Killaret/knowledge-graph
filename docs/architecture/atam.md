# ATAM анализ — Knowledge Graph

**Версия:** 1.0  
**Дата:** Март 2026  
**Метод:** Architecture Tradeoff Analysis Method (сценарный подход)

---

## 1. Введение

ATAM применялся для выявления компромиссов между качественными характеристиками архитектуры. Анализировались три ключевых сценария:

1. **Создание заметки с большим содержимым**
2. **Получение рекомендаций для заметки с 1000+ связей**
3. **Удаление заметки с 5000+ входящими ссылками**

---

## 2. Сценарий 1: Создание заметки с большим содержимым

### 2.1. Описание
Пользователь создаёт заметку с текстом >10 000 слов. Система должна сохранить заметку, извлечь ссылки и пересчитать эмбеддинги.

### 2.2. Качественные характеристики
- **Производительность:** время ответа API (пользователь не ждёт тяжёлых операций).
- **Надёжность:** все асинхронные задачи должны быть выполнены (или повторены при сбое).

### 2.3. Решение
- Синхронно: валидация, сохранение заметки, возврат `201 Created`.
- Асинхронно (через очередь asynq): извлечение ссылок, пересчёт эмбеддингов.

### 2.4. Компромиссы
| За | Против |
|----|--------|
| Пользователь не ждёт (быстрый ответ) | Связи и эмбеддинги появятся не сразу (eventual consistency) |
| Очередь обеспечивает надёжность (повторы) | Нужна инфраструктура (Redis, воркеры) |
| Легко масштабировать воркеры | При падении воркера — устаревшие данные |

### 2.5. Принятое решение
Оставить асинхронную обработку с очередью asynq. При сбое задачи — повтор через `Retry`.

---

## 3. Сценарий 2: Получение рекомендаций (заметка с 1000+ связей)

### 3.1. Описание
Пользователь запрашивает рекомендации для заметки, имеющей 1000 прямых и косвенных связей. Система должна выполнить BFS глубиной 3 и вернуть топ-10 похожих заметок.

### 3.2. Качественные характеристики
- **Производительность:** время ответа < 500 мс (в идеале).
- **Масштабируемость:** работа при росте числа заметок до 1 млн.

### 3.3. Решение
- Ограничение первого уровня до 50 кандидатов.
- Кэширование результатов в Redis (TTL = 5 минут).
- BFS с затуханием λ=0.5, глубина 3.

### 3.4. Компромиссы
| За | Против |
|----|--------|
| Кэш даёт мгновенный ответ для повторных запросов | Устаревшие данные (TTL 5 мин) |
| Ограничение кандидатов контролирует нагрузку | Можно упустить важные связи (если они за пределами топ-50) |
| BFS на уровне БД (рекурсивные CTE) эффективен | При 1 млн заметок потребуется оптимизация индексов |

### 3.5. Принятое решение
- Использовать кэширование (Redis) + ограничение первого уровня (50).
- При промахе кэша выполнять BFS через рекурсивные CTE PostgreSQL.
- Планировать переход на предвычисление рекомендаций при росте графа.

## 4. Сценарий 3: Удаление заметки с 5000+ входящими ссылками

### 4.1. Описание
Пользователь удаляет заметку, на которую ссылаются 5000 других заметок. Нужно удалить саму заметку и все связанные с ней связи.

### 4.2. Качественные характеристики
- **Надёжность:** данные не должны теряться (целостность).
- **Производительность:** операция не должна блокировать другие запросы.

### 4.3. Решение (MVP)
- Синхронное удаление в одной транзакции (каскадное).
- Использовать индексы на `source_id` и `target_id` для быстрого удаления.
- При превышении порога (>5000 связей) — перейти на асинхронное удаление через outbox.

### 4.4. Компромиссы
| За | Против |
|----|--------|
| Простота реализации | Долгое выполнение при большом количестве связей |
| Транзакционная целостность | Блокировка таблицы на время удаления |
| Индексы ускоряют операцию | При 10 000+ связей возможны таймауты |

### 4.5. Принятое решение
- Для MVP — синхронное удаление (пока связи <5000).
- Добавить индексы на `(source_id, target_id)`.
- При достижении порога — перейти на асинхронное удаление + outbox.

## 5. Итоговая таблица компромиссов

| Сценарий | Ключевой компромисс | Выбранное решение |
|----------|---------------------|-------------------|
| Создание заметки | Скорость ответа vs актуальность связей | Асинхронная очередь (eventual consistency) |
| Рекомендации | Свежесть vs производительность | Кэширование (TTL 5 мин) + ограничение кандидатов |
| Удаление заметки | Простота vs масштабируемость | Синхронное удаление до 5000 связей, затем асинхронное |

## 6. Неанализированные сценарии (отложены)

- Отказ сервиса (горизонтальное масштабирование)
- Рост данных (1 млн заметок)

## 7. Выводы

Архитектура Knowledge Graph обеспечивает баланс между простотой (MVP) и масштабируемостью (асинхронные задачи, кэширование, пороги). Основные компромиссы осознанно приняты и документированы.
