# Архитектурные решения (ADR)

**Версия:** 1.0  
**Дата:** Март 2026

---

## ADR-001: DDD с чёткими слоями

**Контекст:** Система должна быть гибкой, тестируемой и независимой от внешних инфраструктурных деталей.

**Решение:** Использовать Domain-Driven Design с четырьмя слоями:
- **Domain** — бизнес-логика, не зависит от внешних систем.
- **Application** — сценарии использования (use cases), оркестрация.
- **Infrastructure** — адаптеры к БД, очередям, внешним сервисам.
- **Interfaces** — HTTP handlers, DTO.

**Последствия:**
- ✅ Чёткое разделение ответственности.
- ✅ Domain легко тестировать изолированно.
- ❌ Увеличивается количество файлов и интерфейсов.

## ADR-002: CQRS (команды и запросы)

**Контекст:** Операции изменения состояния и чтения данных имеют разные требования к производительности и масштабированию.

**Решение:** Разделить операции на **Commands** (изменяют состояние) и **Queries** (только читают). Использовать in-memory Command Bus и Query Bus.

**Последствия:**
- ✅ Чёткое разделение, легче тестировать.
- ✅ Позволяет в будущем добавить read model.
- ❌ Небольшое усложнение кода (дополнительные интерфейсы).

## ADR-003: Link — отдельная сущность

**Контекст:** Заметка может иметь тысячи связей. Загрузка всех связей при каждом запросе заметки неэффективна.

**Решение:** Хранить связи в отдельной таблице/агрегате `Link`. Получать связи через отдельный запрос с пагинацией.

**Последствия:**
- ✅ Производительность (не грузим все связи).
- ✅ Гибкость (пагинация, фильтрация).
- ❌ Требуется отдельный репозиторий для связей.

## ADR-004: Синхронное удаление связей (MVP)

**Контекст:** При удалении заметки нужно удалить все связанные с ней связи (каскадно). В MVP ожидается до 1000 связей на заметку.

**Решение:** Удалять связи синхронно в той же транзакции, что и заметку. При росте числа связей (>5000) перейти на асинхронное удаление через outbox.

**Последствия:**
- ✅ Простота реализации.
- ✅ Гарантированная целостность (транзакция).
- ❌ Потенциальное замедление при большом количестве связей.

## ADR-005: Валидация — Value Objects + Application

**Контекст:** Данные должны быть валидными на всех уровнях системы.

**Решение:**
- **Базовые проверки** (длина, формат) — в Value Objects (Domain).
- **Бизнес-правила** (циклы в графе) — в доменных сервисах.
- **Проверка существования** — в Application слое (через репозиторий).

**Последствия:**
- ✅ Нельзя создать невалидный объект.
- ✅ Чёткое распределение ответственности.
- ❌ Немного больше кода для Value Objects.

## ADR-006: Eventual consistency для read model

**Контекст:** Для проверки существования ID при создании связей можно использовать отдельную read model (таблицу идентификаторов). Задержка в несколько миллисекунд не критична.

**Решение:** Использовать eventual consistency для read model. При создании заметки асинхронно обновлять таблицу существующих ID.

**Последствия:**
- ✅ Упрощает валидацию (быстрый поиск по ID).
- ✅ Снижает нагрузку на основную таблицу.
- ❌ Возможна кратковременная несогласованность (окно в несколько мс).

## ADR-007: Value Objects (Title, Content, LinkType)

**Контекст:** Примитивные поля (string, int) не гарантируют корректность данных и не выражают бизнес-смысл.

**Решение:** Использовать Value Objects: `Title`, `Content`, `LinkType`, `Weight`. Каждый VO сам проверяет свою корректность при создании.

**Последствия:**
- ✅ Безопасность типов.
- ✅ Самодокументируемый код.
- ❌ Небольшое увеличение кода.

---

## ADR-008: Эмбеддинги в Infrastructure, Core не знает о векторах

**Контекст:** Эмбеддинги — техническая деталь для поиска, не бизнес-сущность.

**Решение:** Хранить эмбеддинги в отдельной таблице `note_embeddings` (pgvector). Репозиторий эмбеддингов — в Infrastructure слое. Core работает только с доменными сущностями.

**Последствия:**
- ✅ Core не зависит от технических деталей (pgvector, float32).
- ✅ Легко заменить хранилище эмбеддингов.
- ❌ При запросе рекомендаций нужен дополнительный вызов к Infrastructure.

## ADR-009: Domain Events + Event Bus (Application)

**Контекст:** При изменении состояния системы (создание заметки, добавление связи) нужно асинхронно реагировать (пересчёт эмбеддингов, уведомления).

**Решение:** Генерировать доменные события (NoteCreated, LinkAdded). Публиковать их через Event Bus в Application слое. Подписчики запускают асинхронные задачи.

**Последствия:**
- ✅ Слабая связанность.
- ✅ Асинхронная реакция не блокирует основной поток.
- ❌ Усложнение отладки (потоки событий).

## ADR-010: Миграции (папка migrations/)

**Контекст:** Схема БД должна версионироваться и воспроизводиться на разных окружениях.

**Решение:** Хранить SQL-миграции в папке `backend/migrations/`. Использовать `golang-migrate` для применения миграций.

**Последствия:**
- ✅ Версионирование схемы.
- ✅ Простота развёртывания на новом окружении.
- ❌ Требуется осторожность при написании миграций (обратная совместимость).

## ADR-011: Specifications в Domain слое

**Контекст:** Сложные фильтрации (по тегам, датам, ключевым словам) должны быть переиспользуемыми и выражать бизнес-правила.

**Решение:** Внедрить паттерн Specification в Domain слое. Каждая спецификация реализует методы `IsSatisfiedBy` (проверка в памяти) и `ToSQL` (генерация SQL для репозитория).

**Последствия:**
- ✅ Переиспользование логики фильтрации.
- ✅ Можно комбинировать (And, Or, Not).
- ❌ Усложнение кода (дополнительные интерфейсы и комбинаторы).

## ADR-012: In-memory Bus (без middleware)

**Контекст:** Для MVP не нужна сложная диспетчеризация с middleware (логирование, метрики).

**Решение:** Использовать простой in-memory CommandBus и QueryBus с регистрацией команд/запросов через map[reflect.Type].

**Последствия:**
- ✅ Простота реализации.
- ✅ Легко добавить middleware позже.
- ❌ Отсутствует встроенное логирование/мониторинг (добавим позже).

## ADR-013: Эмбеддинги хранятся отдельно (таблица note_embeddings)

**Контекст:** Эмбеддинги занимают ~1.5 КБ на заметку. Для 1 млн заметок это ~1.5 ГБ + индексы.

**Решение:** Хранить эмбеддинги в отдельной таблице `note_embeddings` с полем `embedding vector(384)`. Использовать индекс IVFFlat для быстрого поиска.

**Последствия:**
- ✅ Отдельное управление памятью и индексами.
- ✅ Можно не загружать эмбеддинги при обычных запросах.
- ❌ Дополнительный JOIN при запросах, где нужны эмбеддинги.

## ADR-014: Progressive Rendering (Fog of War) для 3D графа

**Контекст:** Загрузка больших графов (1000+ узлов) может вызывать задержки и перегрузку GPU. Нужен способ плавной загрузки с визуальной обратной связью.

**Решение:** Использовать **Progressive Rendering** с эффектом "Fog of War":
- Узлы загружаются инкрементально (партиями по N штук)
- Туман (Three.js FogExp2) изначально плотный и разрежается по мере загрузки
- Камера анимируется автоматически для показа новых узлов
- Каждый узел появляется с анимацией (opacity + position)

**Технические детали:**
- Three.js модульная архитектура: `core/`, `simulation/`, `rendering/`, `camera/`
- d3-force-3d для физики симуляции
- `addNodesToSimulation()` - добавление узлов с анимацией
- `lerpCamera()` - плавные переходы камеры
- `animateFogDensity()` - анимация тумана

**Последствия:**
- ✅ Плавный UX при загрузке больших графов
- ✅ Визуальная обратная связь о прогрессе загрузки
- ✅ Возможность отмены/прерывания загрузки
- ❌ Усложнение фронтенд-архитектуры (дополнительные модули)
- ❌ Требуется производительный GPU для 3D режима
