# Спецификация: Иерархическая кластеризация ключевых слов для зум-визуализации

**Версия:** 1.0  
**Дата:** 2026-04-21  
**Статус:** Требуется реализация (Backend)

---

## 1. Концепция и цели

Построение динамической перекрывающейся иерархии (soft clustering) ключевых слов для многоуровневой визуализации графа знаний. При изменении масштаба (zoom) пользователь видит агрегированные тематические группы с их названиями и пространственными границами, сформированные на основе семантической близости и совместной встречаемости ключевых слов в заметках.

### Ключевые требования

- Поддержка нескольких уровней иерархии (до 4-5)
- Предрасчёт пространственных данных кластеров (центроид, радиус) для быстрой отрисовки
- Версионирование состояния кластеризации для стабильности пользовательского интерфейса
- Возможность ручного переопределения названий кластеров

## 2. Критические архитектурные риски и контрмеры

### 2.1. Взрыв джойнов при рендеринге

**Риск:** Выполнение агрегации весов и расчёт геометрии на лету при каждом запросе `/clusters/by-zoom` приведёт к деградации производительности при >100 заметок в области видимости.

**Решение:**
- Ввести материализованную таблицу `cluster_spatial`, содержащую предрассчитанные координаты, радиусы и топ-ключевые слова для каждого кластера на каждом уровне
- API `/clusters/by-zoom` выполняет один запрос к этой таблице с фильтрацией по уровню и, опционально, пространственному пересечению с bounds
- Никаких JOIN с `notes` или `note_cluster_weights` во время запроса

### 2.2. Конфликт динамичности и стабильности кадра

**Риск:** Фоновая перекластеризация изменяет структуру и координаты групп, разрушая ментальную карту пользователя между сессиями.

**Решение:**
- Версионирование дерева кластеров (поле `version` в таблицах)
- API принимает параметр `?version=N`. По умолчанию отдаётся последняя версия
- Эндпоинт `GET /clusters/version` возвращает текущую версию и флаг устаревания
- Фронтенд кеширует версию на время сессии и предлагает пользователю явно обновить карту при появлении новой версии
- Для анимации перехода предоставляется эндпоинт `GET /clusters/diff` с изменёнными координатами

### 2.3. Семантический шум в лейблах

**Риск:** Автоматические имена кластеров на основе частотных слов превращаются в нечитаемые наборы тегов.

**Решение:**
- Двухуровневая система: автоматическое имя (TF-IDF по множеству заметок кластера) и пользовательский алиас (`user_label`)
- В API всегда отдаётся либо `user_label` (если задан), либо автоматическое имя
- Возможность ручного переименования через административный API

## 3. Схема базы данных (дополнения к существующей)

### 3.1. Таблица `keyword_embeddings` (кеш векторов слов)

```sql
CREATE TABLE keyword_embeddings (
    keyword VARCHAR(255) PRIMARY KEY,
    embedding vector(384) NOT NULL,
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);
```

### 3.2. Таблица `keyword_clusters` (версионированное дерево)

```sql
CREATE TABLE keyword_clusters (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    version INT NOT NULL,
    tree JSONB NOT NULL,          -- полное дерево (формат см. п. 3.3)
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);
CREATE INDEX idx_keyword_clusters_version ON keyword_clusters(version DESC);
```

### 3.3. Формат JSONB-дерева

```json
{
  "id": "uuid-root",
  "stable_id": "md5-hash",
  "name": "Все темы",
  "level": 0,
  "user_label": null,
  "children": [
    {
      "id": "uuid-1",
      "stable_id": "md5-hash",
      "name": "Наука",
      "level": 1,
      "user_label": null,
      "keywords": ["физика", "химия", "биология"],
      "children": [...]
    }
  ]
}
```

Поле `stable_id` обязательно, генерируется детерминированно (например, `md5(name + sorted(keywords))`).

### 3.4. Таблица `note_cluster_weights` (быстрая связь заметка-кластер)

```sql
CREATE TABLE note_cluster_weights (
    note_id UUID NOT NULL REFERENCES notes(id) ON DELETE CASCADE,
    cluster_id UUID NOT NULL,     -- соответствует id в JSONB-дереве
    level INT NOT NULL,
    weight FLOAT NOT NULL,
    version INT NOT NULL,         -- версия кластеризации
    PRIMARY KEY (note_id, cluster_id, version)
);
CREATE INDEX idx_note_cluster_weights_cluster_version ON note_cluster_weights(cluster_id, version);
```

### 3.5. Таблица `cluster_spatial` (материализованная геометрия)

```sql
CREATE TABLE cluster_spatial (
    cluster_id UUID NOT NULL,           -- соответствует id в JSONB-дереве
    version INT NOT NULL,
    level INT NOT NULL,
    center_x FLOAT NOT NULL,
    center_y FLOAT NOT NULL,
    radius FLOAT NOT NULL,              -- среднеквадратичное отклонение координат заметок
    bbox_xmin FLOAT,
    bbox_ymin FLOAT,
    bbox_xmax FLOAT,
    bbox_ymax FLOAT,
    top_keywords TEXT[] NOT NULL,       -- топ-5 слов для тултипов
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    PRIMARY KEY (cluster_id, version)
);
CREATE INDEX idx_cluster_spatial_level_version ON cluster_spatial(level, version);
```

**Важно:** Координаты заметок берутся из `notes.metadata` (поля `position.x`, `position.y`). Предполагается, что заметки имеют пространственные координаты, задаваемые при создании/редактировании или вычисляемые графовым layout.

## 4. Алгоритм итеративной кластеризации

### 4.1. Комбинированная метрика сходства

Для пары элементов (слово или кластер) сходство вычисляется как:

```
similarity = λ * cosine(emb(A), emb(B)) + (1-λ) * jaccard_weighted(notes(A), notes(B))
```

- `λ` — вес семантической компоненты (рекомендуемое значение 0.7)
- Эмбеддинги берутся из `keyword_embeddings` (если отсутствуют — косинусная часть равна 0)
- Jaccard с учётом весов ключевых слов в заметках

### 4.2. Итеративное построение уровней

**Уровень 0:** Каждое уникальное ключевое слово образует отдельный элемент с множеством заметок, в которых оно встречается.

Для каждого следующего уровня:
1. Строится матрица попарного сходства для элементов предыдущего уровня
2. На основе порога `threshold` (рекомендуемое 0.2) формируется граф, компоненты связности которого становятся новыми кластерами
3. Для каждого нового кластера агрегируются: множество заметок (сумма весов), эмбеддинг (среднее арифметическое), список исходных ключевых слов
4. Остановка при достижении `maxLevels` (по умолчанию 4) или если ни один кластер не объединился

### 4.3. Построение дерева и заполнение `cluster_spatial`

После завершения итераций:
1. Строится JSONB-дерево с вычислением `stable_id` для каждого узла
2. Для каждого узла рассчитываются пространственные характеристики на основе координат заметок, входящих в кластер (через `note_cluster_weights`)
3. Данные сохраняются в `cluster_spatial` с новой версией

## 5. API спецификация

### 5.1. GET /clusters/version

Возвращает информацию о текущей версии кластеризации.

**Ответ:**
```json
{
  "version": 5,
  "updated_at": "2026-04-21T03:00:00Z",
  "stale": false
}
```

### 5.2. GET /clusters/by-zoom

Возвращает кластеры для заданного масштаба и (опционально) области видимости.

**Параметры:**
- `zoom` (float, 0..1) — нормализованный уровень приближения
- `bounds` (object, опционально) — `{xmin, ymin, xmax, ymax}`
- `version` (int, опционально, default = latest) — версия данных

**Логика определения уровня:**
```
level = floor((1 - zoom) * maxLevel)
```
где `maxLevel` — максимальный уровень в текущей версии дерева.

**Ответ:**
```json
[
  {
    "id": "md5-hash-123",
    "name": "Машинное обучение",
    "level": 2,
    "weight": 42.5,
    "note_count": 12,
    "center": {"x": 120.0, "y": 300.0},
    "radius": 45.0,
    "keywords": ["нейросеть", "обучение", "данные"]
  }
]
```

### 5.3. GET /clusters/tree

Возвращает полное дерево кластеров (для администрирования или кеширования).

**Параметры:**
- `version` (опционально, default = latest)

**Ответ:** JSONB-дерево в том виде, как хранится в БД.

### 5.4. GET /clusters/diff

Возвращает изменения координат кластеров между двумя версиями.

**Параметры:**
- `from_version` (int)
- `to_version` (int)

**Ответ:**
```json
[
  {
    "id": "md5-hash-123",
    "old_center": {"x": 110.0, "y": 290.0},
    "new_center": {"x": 120.0, "y": 300.0}
  }
]
```

### 5.5. POST /admin/clusters/rebuild

Запускает полную перекластеризацию (асинхронно через asynq).

**Параметры (опционально):**
```json
{
  "lambda": 0.7,
  "threshold": 0.2,
  "max_levels": 4
}
```

**Ответ:**
```json
{
  "task_id": "uuid",
  "status": "accepted"
}
```

### 5.6. PATCH /admin/clusters/{id}/label

Ручное переименование кластера.

**Тело запроса:**
```json
{
  "user_label": "Глубокое обучение"
}
```

## 6. Стратегия обновлений

### 6.1. Инкрементальное обновление (при добавлении заметки)

Для каждого ключевого слова новой заметки определяется его путь в текущем дереве (поиск по полю `keywords` листьев).

Для всех узлов на этом пути добавляются записи в `note_cluster_weights` с текущей версией.

Структура дерева и `cluster_spatial` не изменяются.

### 6.2. Полная перекластеризация

- Запускается автоматически раз в сутки (например, в 03:00) через asynq-периодическую задачу
- Может быть запущена вручную через административный эндпоинт
- Создаёт новую версию данных в `keyword_clusters`, `note_cluster_weights` (для новой версии) и `cluster_spatial`
- Старые версии сохраняются для возможности отката и diff-запросов (хранить не более 3 последних версий)

### 6.3. Очистка старых версий

При создании новой версии удалять данные версий старше N (например, старше 3). Это предотвращает неконтролируемый рост таблиц.

## 7. Рекомендации по реализации

- **Язык:** Go 1.22+
- **Сервис:** `ClusterService` в `backend/internal/service/cluster_service.go`
- **Репозитории:** `KeywordEmbeddingRepository`, `ClusterRepository`, `NoteClusterWeightRepository`, `ClusterSpatialRepository`
- **Asynq задача:** `tasks.ClusterRebuildTask`
- **Периодический запуск:** через `asynq.Scheduler`

### Тестирование

- Модульные тесты на функцию комбинированного сходства
- Интеграционный тест полного цикла: создание заметок → перекластеризация → проверка API
- Нагрузочное тестирование эндпоинта `/clusters/by-zoom` с 1000+ заметок

> **Примечание:** Для ускорения кластеризации на больших объёмах рассмотреть использование приближённого поиска ближайших соседей через pgvector (если слов > 10000).
